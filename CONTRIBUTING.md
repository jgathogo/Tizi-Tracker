## How to Report Issues

### Before Reporting

1. **Check existing issues** - Your issue might already be reported
2. **Search closed issues** - It might have been fixed already
3. **Check the README** - Your question might be answered there

### Creating an Issue

Use the [GitHub Issues](https://github.com/jgathogo/Tizi-Tracker/issues) page and include:

1. **Clear title** - What's the problem?
2. **Description** - What happened vs. what you expected
3. **Steps to reproduce** - How can we reproduce it?
4. **Environment:**
   - Browser (Chrome, Safari, Firefox, etc.)
   - Device (Desktop, iPhone, Android)
   - App version (if applicable)
5. **Screenshots** - If visual issues
6. **Console errors** - Open DevTools (F12) and check Console tab

## Suggesting Enhancements

### Feature Requests

We welcome feature suggestions! When proposing a feature:

1. **Check existing issues** - Someone might have suggested it
2. **Explain the use case** - Why would this be useful?
3. **Describe the solution** - How should it work?
4. **Consider alternatives** - Are there simpler ways to achieve this?

## Development Setup

If you want to contribute code:

1. **Fork the repository**
2. **Clone your fork:**
   ```bash
   git clone https://github.com/YOUR_USERNAME/Tizi-Tracker.git
   cd Tizi-Tracker
   ```

3. **Install dependencies:**
   ```bash
   npm install --legacy-peer-deps
   ```

4. **Start dev server:**
   ```bash
   npm run dev
   ```

5. **Create a branch:**
   ```bash
   git checkout -b feature/your-feature-name
   ```

6. **Make your changes** and test them

7. **Commit with clear messages:**
   ```bash
   git commit -m "Add: Feature description"
   ```

8. **Push and create Pull Request:**
   ```bash
   git push origin feature/your-feature-name
   ```

## Code Style

- **TypeScript** - Use type hints
- **React Hooks** - Prefer functional components
- **Tailwind CSS** - Use utility classes
- **Naming** - Clear, descriptive names
- **Comments** - Explain "why", not "what"

## Pull Request Guidelines

1. **One feature per PR** - Keep it focused
2. **Update README** - If adding features or changing setup
3. **Test your changes** - Make sure it works
4. **Describe your changes** - What did you add/fix?
5. **Reference issues** - Link to related issues (e.g., "Fixes #123")

## Questions?

- Open a **Question** issue on GitHub
- Check existing discussions in Issues
